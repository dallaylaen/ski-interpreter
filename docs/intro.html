Simple Kombinator Interpreter Intro

A bit of history

Combinatory logic is a minimalistic functional programming language.

It was discovered in 1920 by Moses Schönfinkel as a way to drive lexical variables out of logical expressions (hence "logic").

Little did he know that the tool is in fact too powerful for the task. Neither Gödel's incompleteness theorem, nor the halting problem were known at the time.

The research was continued by Haskell Curry, Alonzo Church, and others, influencing the development of lambda calculus and functional programming languages such as Lisp and Haskell.

Although combinators turned out to be too cumbersome for direct use in practical programming, they do appear here and there, sometimes in disguise.

How computations work

The following definitions are informal and lack rigor, but hopefully are good enough to provide some insights.

A combinatory logic expression consists of a starting term, followed by zero or more expressions (arguments). An application of two expressions is performed by simply appending the latter in its entirety (in parentheses, if needed) to the former's argument list.

By this definition, application is left-associative: (a b) c is the same as a b c or a(b)(c), but not a (b c).

Similarly, applying a b to c d yields a b (c d) - or equivalently a(b)(c(d)) - but not a b c d or a(b(c(d))).

A combinator is a function with fixed number of arguments that uses only application on said arguments and possibly some other previously defined terms. Arguments may be combined, reordered, duplicated, or discarded in doing so. Combinators are closed under application (this requires proving).

If the starting term of a (sub-)expression is a combinator and is followed by enough arguments, it can be reduced, that is, the starting term and its arguments are replaced by whatever expression it produces, and the rest of the expressions (if any) are simply applied at the end as is. This process can be iterated as long as a reduction can be made, and that's how computations actually work.

If an expression has no possible reductions, it is said to be in normal form. If multiple terms may be reduced at the same time - which is common for complex expressions, - not all reduction orders may lead to termination. However, if two different orders lead to some normal form, it is always the same normal form. (That's Church - Rosser's theorem). In particular, always going for the leftmost eligible term guarantees reaching the normal form (if it exists at all) and is called the normal reduction strategy.

The SKI system

Now with this bit of theory, let's define some combinators and fool around.

The simplest of all is I, the identity: I x = x. It only has one argument, and returns that argument, unmodified. Looks pointless but it's vital for transitions between other, more complex terms.

The K combinator, or constant, has two arguments: K x y = x. As you can see, it discards the second argument. Note that K x is itself a combinator with just one argument that always returns x.

The S combinator has three arguments: S f g x = f(x)(g(x)). Mind boggling! It can perhaps be broken down a bit:
S x y z = {
  let y = f(x);
  let z = g(x);
  return y(z);
}

Let's see how the three work together. We'll use bold text for terms eligible for reduction.

SII(SII)
I(SII)(I(SII))
SII(I(SII))
SII(SII)
...
Oops... We've reached the starting expression! So here is an infinite loop.

S(K(SI))K x y
K(SI)x(K x)y
SI(K x)y
Iy(K x y)
y(K x y)
y x
The terms x and y just got swapped!

Incidentally, the above expression is called the T combinator: T x y = y x. Just like with K x, the term T x can be viewed as a promise waiting for a callback (y).

SKKx
Kx(Kx)
x
And we just emulated I using S and K.

An attempt at a better definition

Just like T and K, any combinator of arity n plus a variable is a combinator of arity n-1, per definition (it applies its arguments to themselves and possible some previously defined terms). So a combinator may be thought of as a one-argument function returning a combinator or an application thereof.

Similarly, since application is left-associative, we might as well say it's a binary operation. The expression tree then becomes a binary tree, and the root value becomes the leftmost leaf value. (Picture pending). So, these rules can be formally summarized as

expr ::= var | expr expr | λ var . expr

which is, not quite surprisingly, the definition of lambda calculus.

So in fact any combinatory logic expression may be expressed via lambda calculus. The opposite, however, is only true for specific set of starting terms, or bases. In particular, {S, K} is such base.

The interpreter

This interpreter's syntax is as follows:

  • Unknown terms are assumed to be free variables.
  • Uppercase terms like S, K, B are always one-letter and can be lumped together.
  • Lowercase terms like x, y, pred, list, foo_42 are considered words/identifiers and must be spaced from one another. Hopefully this convention is equally annoying to both programmers and mathematicians.
  • When displaying expressions, parentheses are omitted where possible, although it may not be beginner friendly. Perhaps in some future version it will be possible to cycle between lisp-like, C-like, and terse notation. Of course, the parser accepts as many parentheses as one wishes: (((( ((((x)))) ((((y)))) )))) is still a valid expression and is still equal to x y.
  • New terms may be defined as name = impl, where impl is a valid expression. Multiple definitions may be given, separated by semicolon; in this case only the last one will be executed and/or added to the term inventory. The preceding definitions will be localized to the last expression.
  • Lambdas are written as var -< impl and are left associative: T = x->y->y x. Lambdas are lazy: the implementation will not be reduced whatsoever until the (last) variable is substituted.
  • A nonnegative integer is assumed to be a Church numeral: n f x = f(f(... {n times} ...f(x)...)). Church numerals must be separated by spaces from both uppercase and lowercase terms. A special + operator increases the next term by 1 if it happens to be a Church numeral: 7 + 0 = +(+(+(+(+(+(+(0))))))) = +(+(+(+(+(+(1)))))) = ... = 7. This can thus be used to convert calculated numbers to canonical form.
  • By default, the following terms are available:
    • B x y z = x (y z);
    • C x y z = x z y;
    • I x = x;
    • K x y = x;
    • S f g x = f x (g x);
    • W f x = f x x;

Connection to lambda calculus

An arbitrary lambda expression can be converted to SKI via the following rules:

  • x ↦ x becomes I;
  • x ↦ f(x) (where f is independent of x) becomes f;
  • x ↦ y (where y does not depend on x) becomes K y;
  • x ↦ y(z) becomes S(x ↦ y)(x ↦ z), where y and z may or may not depend on x.
  • The process is repeated from the innermost lambda expression outwards and is guaranteed to terminate. (proof?..)

As a matter of fact, I can be derived from S and K and those two thus form a basis together.
